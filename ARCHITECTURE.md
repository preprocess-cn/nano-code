# Architecture

This file describes the architecture of nano-code for LLM consumption.
It focuses on component boundaries, interfaces, data flow, and design rationale.
Human readers may find it useful too, but the primary audience is AI agents editing this codebase.

---

## 1. System Overview

nano-code is a plugin-driven CLI AI coding assistant. The architecture layers from
outermost (user-facing) to innermost (core):

```
CLI Layer (cac arg parser)
  → Orchestration Layer (index.ts: startCLI, main loop)
    → Agent Layer (agent.ts: NanoCodeAgent, ReAct loop)
      → PluginSystem Layer (plugin.ts: PluginRegistry, NanoPlugin)
        → LLM Layer (llm.ts: LLMClient, streaming API)
          → External API (OpenAI-compatible)
```

Supporting systems that span layers:
- DisplayPlugin System (display.ts) — UI output, independent of PluginRegistry
- LogPlugin System (logger.ts) — diagnostic output, independent of PluginRegistry
- Config System (config.ts) — multi-source config layering

## 2. LogPlugin System (logger.ts)

### Purpose
Provide structured diagnostic logging that is decoupled from both the core
(PluginRegistry) and the UI (DisplayPlugin). Multiple backends can receive
log entries simultaneously.

### Key Types

```typescript
type LogLevel = 'debug' | 'info' | 'warn' | 'error';

interface LogEntry {
  timestamp: string;       // ISO-8601, generated by LogManager
  level: LogLevel;
  module: string;          // source module name, e.g. 'main', 'registry'
  message: string;         // log message text
  args?: unknown[];        // additional structured data
  error?: Error;           // extracted from args if last arg is Error
}

interface LogPlugin {
  name: string;            // unique identifier
  onLog?(entry: LogEntry): void;  // hook, called for every log entry
}
```

### LogManager

```
LogManager (singleton via export const logManager)
  .register(plugin)   → adds a LogPlugin to dispatch list
  .unregister(name)   → removes a LogPlugin
  .debug(module, msg, ...args)
  .info(module, msg, ...args)
  .warn(module, msg, ...args)
  .error(module, msg, ...args)
```

Dispatch behavior:
- Creates a LogEntry with current timestamp
- Extracts trailing Error arg into entry.error, rest into entry.args
- Iterates all registered plugins, calling onLog?.(entry) on each
- If no plugins registered, entry is silently dropped

### Default Plugin

```
StderrLogPlugin (name='stderr', auto-registered at module load)
  .onLog(entry):
    writes: `[timestamp] [LEVEL] [module] message` to process.stderr
    if entry.error exists: appends error.stack on next line
```

### Relationship to DisplayPlugin

LogPlugin and DisplayPlugin are independent plugin systems connected by
a bridge in the orchestration layer (index.ts):

```
index.ts startCLI():
  logManager.register({
    name: 'display-bridge',
    onLog(entry) {
      if entry.level === 'error':
        displayMgr.onError(...)
    }
  })
```

This bridge is the ONLY connection between the two systems. It lives in the
orchestration layer, not in either plugin system.

### Relationship to PluginRegistry (Core)

PluginRegistry does NOT use LogManager. All PluginRegistry diagnostics use
console.warn/console.error directly. This is intentional:
- The core should not depend on any plugin system
- console.* is the universal lowest-common-denominator
- Users who want structured logging register LogPlugins independently

```
PluginRegistry → console.error(...) → process.stderr
Core modules   → logManager.error(...) → StderrLogPlugin → stderr
                                       → DisplayBridge  → DisplayManager
```

### Data Flow Diagram

```
[Src index.ts error handler]
  → logManager.error('main', msg, error)
    → LogManager._log('error', 'main', msg, error)
      → LogEntry { timestamp, level:'error', module:'main', message, error }
      → StderrLogPlugin.onLog(entry)    → process.stderr.write(...)
      → DisplayBridge.onLog(entry)      → displayMgr.onError(...)

[PluginRegistry internal error]
  → console.error('[plugin] ...', err)  → process.stderr.write(...)
```

Key insight: PluginRegistry's console.error output goes ONLY to stderr.
If a user wants PluginRegistry errors to appear in the Ink UI, they would
need to register a LogPlugin in PluginRegistry — but this is intentionally
not done by default to keep the core clean.

## 3. DisplayPlugin System (display.ts)

### Purpose
Manage user-facing UI output. Multiple display backends (REPL, Ink) can be
registered, but typically only one is active at a time.

### Key Types

```typescript
export type MessageLevel = 'status' | 'info' | 'warn' | 'error' | 'success';

interface AgentEvent {
  agentName: string;
}

interface StatusEvent extends AgentEvent {
  message: string;
  level: MessageLevel;  // 展示层据此决定渲染样式
}

interface DisplayPlugin {
  name: string;
  ownsOutput?: boolean;        // 是否独占全部终端输出
  rawInput?: boolean;          // 是否需要按键级原始输入
  onInit?(registry): Promise<void>;
  start?(config: StartConfig): void;
  stop?(message: string): void;
  prompt?(): Promise<string | null>;
  onUserInput?(input: string, sourcePlugin: string): void;
  onStatus?(event: StatusEvent): void;
  onStreamChunk?(event: { text: string; agentName: string }): void;
  onToolCall?(event: { toolName: string; args: any; agentName: string }): void;
  onToolResult?(event: { status: string; message?: string; agentName: string }): void;
  onError?(event: { message: string; agentName: string; stack?: string }): void;
  onDebug?(event: { data: string; agentName: string }): void;
  onAgentTurnStart?(event: AgentEvent): void;
  onAgentTurnEnd?(event: AgentEvent): void;
  onStateSnapshot?(snapshot: { agentName: string; messageCount: number }): void;
  onContextAnalysis?(analysis: ContextAnalysis): void;
}
```

### DisplayManager

```
DisplayManager
  .addPlugin(plugin)    // adds a display backend
  .init(registry)       // called after plugin registration
  .start(options)       // starts the display
  .stop(message)        // stops the display
  .prompt()             // get user input
  .onStreamChunk(...)   // dispatch to all plugins
  .onToolCall(...)
  .onToolResult(...)
  .onStatus(...)
  .onError(...)
  .onUserInput(...)
  .onAgentTurnStart(...)
  .onAgentTurnEnd(...)
  .onStateSnapshot(...)
```

Dispatch: iterates all added plugins, calls hook if implemented (optional).

## 4. PluginRegistry (plugin.ts)

### Purpose
Core plugin management: registration, tool dispatch, hook chains, lifecycle.

### Key Types

```typescript
interface NanoPlugin {
  name: string;
  description?: string;
  version?: string;
  getTools(): ToolDefinition[];
  execute(name, args, ctx): Promise<ToolResponse>;
  onInit?(registry): Promise<void>;
  onDestroy?(): Promise<void>;
  onSystemPrompt?(prompt: string): string;
  onBeforeRequest?(messages): ChatMessage[];
  onAfterRequest?(response, rawMeta?): void;
  onBeforeToolCall?(toolCall): ToolCall | null;
  onAfterToolCall?(result): ToolResponse;
  onBeforeAgentInput?(input): Promise<CommandInterceptResult | null>;
  onExtraParams?(): Record<string, unknown>;
}
```

### PluginRegistry

```
PluginRegistry
  // Registration
  .register(plugin)          → adds plugin, indexes tools, calls onInit
  .unregister(name)          → calls onDestroy, removes from maps
  .destroy()                 → calls onDestroy on ALL plugins, clears all state

  // Tool dispatch
  .execute(name, args, ctx)  → resolves plugin by tool index, executes with timeout
  .getAllSchemas()           → collects ToolDefinition from all plugins

  // Hook chains (execPipe for transform, execBroadcast for void)
  .execSystemPrompt(prompt)
  .execBeforeRequest(messages)
  .execAfterRequest(response, meta)
  .execBeforeToolCall(toolCall)
  .execAfterToolCall(result)
  .collectExtraParams()
  .execBeforeAgentInput(input)

  // Permission gate
  .allowTool(name)           → add to session allowlist
  .isToolAllowed(name)
  .clearPermissions()
  .getAllowedTools()

  // Config per plugin
  .setPluginConfig(name, config)
  .getPluginConfig(name)

  // Store (shared state channel)
  store: IStore
```

Error handling in PluginRegistry: ALL hook errors are caught and logged via
console.error (never logManager, never throw). This prevents a misbehaving
plugin from crashing the agent loop.

### Destroy Flow

```
handleExit(displayMgr, registry)
  → registry?.destroy()
    → for each plugin: plugin.onDestroy?.()
      → MCP transport: stop() sends SIGTERM, waits 3s, SIGKILL fallback
    → clear all maps, clear allowlist
  → displayMgr?.stop(EXIT_MESSAGE)
  → process.exit(0)
```

## 5. NanoCodeAgent (agent.ts)

### Purpose
ReAct loop: build prompt → call LLM → execute tools → repeat.

### Flow

```
runTask(userPrompt)
  {
    inject userPrompt into messageHistory
    loop:
      1. build system prompt (via prompt.ts + PluginRegistry hooks)
      2. execBeforeRequest (plugin transform hook)
      3. sendSystemMessage (LLM streaming call)
      4. execAfterRequest (plugin broadcast hook)
      5. if stopReason !== 'tool_use': break
      6. for each toolCall:
        a. display.onToolCall (show tool info)
        b. permission gate (sideEffect check + confirmation callback)
        c. registry.execute (tool dispatch with timeout)
        d. display.onToolResult
        e. push result message into history
      7. update store with new message snapshot
      8. continue loop
    return last assistant content
  }
```

### Cancellation

```
Ctrl+C → store.set(AgentCancelled, true) → abort AbortController
  → LLM stream throws AbortError → loop break
  → checked at multiple points:
    - before LLM call
    - after LLM response
    - before each tool call
```

## 6. LLM Client (llm.ts)

### Purpose
OpenAI-compatible streaming API client with retry logic.

### Key Types

```typescript
interface ToolCallDelta {
  index: number;
  id?: string;
  function: { name?: string; arguments?: string };
}

interface AssembledToolCall {
  id: string;
  type: 'function';
  function: { name: string; arguments: string };
}

interface ChatMessage {
  role: 'user' | 'assistant' | 'system' | 'tool';
  content: string | null;
  tool_calls?: AssembledToolCall[];
  tool_call_id?: string;
  name?: string;
}

interface LLMConfig {
  model?: string;
  temperature?: number;
  apiKey?: string;
  baseURL?: string;
}
```

### sendSystemMessage flow

```
sendSystemMessage(messages, tools, onChunk, extraParams, onMeta, signal)
  retry loop (max 3 attempts, exponential backoff):
    1. POST to /chat/completions with stream:true
    2. iterate SSE chunks:
      - accumulate delta content → onChunk(text)
      - accumulate tool_call deltas → assemble by index
    3. on stream complete:
      - merge tool_call deltas into AssembledToolCall[]
      - read x-usage / include_usage → onMeta(meta)
      - return { text, toolCalls, stopReason }
  on transient error (429/5xx/ECONNRESET):
    - wait delay, retry
  on permanent error:
    - throw
```

## 7. Config System (config.ts)

### Layering (high to low priority)

```
1. Shell environment variables
2. $CWD/.env (project local)
3. ~/.nano-code/.env (global fallback)
4. Project .nano-code.yaml
5. Global ~/.nano-code/config.yaml
6. Defaults (DEFAULT_CONFIG)
```

Plus `--profile <name>` overlay:
- `.nano-code/profiles/<name>.json` (project level)
- `~/.nano-code/profiles/<name>.json` (global level)
- Overrides: agent role, plugin enable/disable, plugin settings

### MCP dual-file config

MCP plugins use a separate config channel alongside the main config layering:

```
config.plugins (NanoConfig)  →  control: which MCP servers are enabled
.mcp.json                    →  data: transport config (command/args/url)
```

At startup, `buildMCPPluginsFromConfig()` JOINs both by server name:
enabled MCP declarations missing from `.mcp.json` trigger a startup warning (orphan detection).

### Config interface

```typescript
interface NanoConfig {
  configVersion: number;
  core: { model?, temperature?, apiKey?, baseURL?, maxTokens?, defaultTimeout? };
  agent: { role?, greeting? };
  plugins: Record<string, PluginConfigEntry>;
  display: { plugin?, enabled? };
  systemPrompt: { withTools?, noTools? };
  skills: { disabled?, disableSkillTool? };
  system_plugins: string[];
  env: Record<string, string>;
}
```

## 8. Key Design Decisions

### 8.1 PluginRegistry does NOT use LogManager

Rationale: The core should have zero dependencies on optional plugin systems.
console.* is universally available and sufficient for diagnostics.

Trade-off: PluginRegistry diagnostics are unstructured (no timestamps, no levels).
If structured logging of core events is needed, the caller (orchestration layer)
can wrap PluginRegistry calls with logManager calls.

### 8.2 LogPlugin and DisplayPlugin are independent systems

Rationale: Logging (machine-readable diagnostics) and Display (user-facing UI)
serve different purposes and should evolve independently.

Connection: Only via the display-bridge in index.ts (orchestration layer),
which forwards error-level log entries to DisplayManager.onError.

### 8.3 DisplayPlugin is NOT managed by PluginRegistry

Rationale: PluginRegistry manages tools and hooks for the agent loop.
DisplayPlugin manages UI output and user input. They are orthogonal.

Display plugins are loaded by index.ts based on config.display.plugin
and managed by DisplayManager, not PluginRegistry.

### 8.4 LogManager uses a singleton pattern

Rationale: Logging is a cross-cutting concern accessed from many modules
(index.ts error handlers, potentially agent.ts, etc.). Passing a LogManager
instance through every constructor would add noise.

Trade-off: Global state makes testing harder. Tests should register a
test-only LogPlugin and verify log entries, or unregister stderr plugin.

### 8.5 StatusEvent level protocol

Rationale: Without a `level` field, the display layer must guess how to render
each message based on fragile string matching (`formatStatusText`). Adding
`level` makes the rendering intent explicit.

```typescript
export type MessageLevel = 'status' | 'info' | 'warn' | 'error' | 'success';
```

| Level | Meaning | Display handling |
|-------|---------|------------------|
| `status` | System state signal (`thinking`, `end`) | Translate to UI indicator, no literal display |
| `info` | User-facing informational message | Display as-is, normal style |
| `warn` | Warning (e.g. tool blocked) | Warning style (yellow/⚠) |
| `error` | Lightweight error notification | Error style (red/✗) |
| `success` | Operation completed | Success style (green/✓) |

### 8.6 Each plugin manages its own strings

Rationale: Centralized `display-strings.ts` created a coupling point — every
module that needed a user-facing string had to import from it. By distributing
strings to each consumer module:
- Core emits only diagnostic logs (`[llm]`, `[plugin]` prefixes) — no display strings
- Display plugins own all their rendering text
- Tool plugins own their LLM-visible response strings
- Orchestration layer (index.ts) owns startup/shutdown messages

### 8.7 CLI display as fallback

Rationale: When no interactive display is configured (e.g. pipe mode), a
non-interactive fallback is needed. `cli-display` writes AI responses to
stdout and status/error messages to stderr, with `prompt()` returning null
to signal non-interactive mode.
